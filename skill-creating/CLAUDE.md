# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

`skill-creating` is itself a **skill** — a markdown-driven instruction set that turns an AI assistant into a skill-building coach. The main entry point is `SKILL.md`, which defines a 7-stage onboarding journey. The repo also ships a set of Python/shell scripts and HTML assets used during the onboarding stages.

## Key scripts

All scripts live in `scripts/`. Run them from the repo root.

```bash
# Check/install Python dependencies (markitdown, pdfplumber, pypdf, Pillow)
bash scripts/ensure-deps.sh

# List all installed skills across ~/.claude/skills, ~/.codebuddy/skills
bash scripts/show-skills.sh

# Scaffold a new skill directory with template SKILL.md
python3 scripts/init_skill.py <skill-name> --path <target-skills-dir>

# Evaluate a skill's structure and content quality
bash scripts/evaluate-skill.sh <skill-directory>

# Package a skill folder into a .skill zip for sharing
python3 scripts/package_skill.py <path/to/skill-folder>
# or as a module:
python -m scripts.package_skill <path/to/skill-folder>

# Aggregate benchmark data across iterations (Stage 7)
python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>

# Run description trigger-rate optimization loop (Stage 7)
python -m scripts.run_loop --eval-set <trigger-eval.json> --skill-path <path> --model <model-id> --max-iterations 5 --verbose

# Generate the eval review viewer (Stage 7)
nohup python eval-viewer/generate_review.py <workspace>/iteration-N \
  --skill-name "my-skill" \
  --benchmark <workspace>/iteration-N/benchmark.json \
  > /dev/null 2>&1 &
# Add --previous-workspace for iteration 2+; --static <output_path> for browserless environments
```

## Architecture

```
SKILL.md              # The skill body — 7-stage onboarding instructions for the AI
scripts/              # Python + shell tools used during onboarding stages
agents/               # Sub-agent instruction files loaded on demand during Stage 7
  grader.md           # Grades eval assertions (pass/fail + evidence)
  comparator.md       # Blind A/B comparison of two skill outputs
  analyzer.md         # Pattern analysis of benchmark results
references/
  schemas.md          # Canonical JSON schemas for evals.json, grading.json, benchmark.json, etc.
assets/
  skill-explainer.html   # Interactive visual explainer (opened in Stage 0)
  eval_review.html       # Template for description-optimization review page (Stage 7)
eval-viewer/
  generate_review.py     # Generates the Stage 7 eval review web app
  viewer.html            # The review viewer template
```

## Skill directory conventions

Skills are installed to (in priority order):
- `~/.claude/skills/<name>/` — Claude Code (recommended)
- `~/.codebuddy/skills/<name>/` — Codebuddy
- `.claude/skills/<name>/` or `.codebuddy/skills/<name>/` — project-level

Every skill must have `SKILL.md` with YAML frontmatter:
```yaml
---
name: skill-name          # letters, digits, hyphens only
description: >
  Use when ...            # MUST describe WHEN to use, not HOW it works
---
```

## Stage 7 eval data flow

```
evals/evals.json          → defines test cases (prompts + expectations)
iteration-N/
  eval-ID/
    with_skill/outputs/   → executor output using skill
    old_skill/outputs/    → executor output using previous skill version (or without_skill/)
    eval_metadata.json    → prompt + assertions for this eval
    grading.json          → grader output (expectations[].text/passed/evidence + summary)
    timing.json           → captured from task notification: total_tokens, duration_ms
benchmark.json            → aggregated across all evals (generated by aggregate_benchmark.py)
```

The `benchmark.json` schema is strict — the eval viewer reads exact field names (`configuration` not `config`; `result.pass_rate` not top-level `pass_rate`). Always reference `references/schemas.md`.
