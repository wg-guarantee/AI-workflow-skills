# Legacy System Archaeology

遗留系统知识考古 - 系统化地挖掘和整理遗留系统的隐性知识，转化为可传承的文档资产。

## Quick Start

```bash
# 1. 创建inventory目录
mkdir -p inventory/{business,code,database,api,ui,ops}

# 2. 使用检查清单跟踪进度
cp assets/templates/checklist.md inventory/CHECKLIST.md

# 3. 开始考古工作
# 按照五维方法论逐个维度进行考古
```

## 工作流程

### Phase 1: 准备阶段
- 确认系统访问权限
- 收集现有文档
- 获取测试账号

### Phase 2-6: 五维考古
按顺序进行各维度考古，每个维度完成后标记检查清单。

| 阶段 | 维度 | 参考文档 | 输出 |
|------|------|----------|------|
| 2 | 业务维度 | `references/business.md` | `business/roles.md`, `workflows.md`, `status-machines.md` |
| 3 | 技术维度 | `references/technical.md` | `code/backend.md`, `frontend.md`, `database/` |
| 4 | 接口维度 | `references/api.md` | `api/endpoints.md`, `error-codes.md` |
| 5 | UI维度 | `references/ui.md` | `ui/pages.md`, `ui-spec.md`, `screenshots/` |
| 6 | 运维维度 | `references/ops.md` | `ops/manuals/`, `test-reports/` |

### Phase 7: 补充和审查
- 生成术语表和FAQ
- 生成README索引
- 质量检查和交付

## 各维度考古指南

### 业务维度考古 → `references/business.md`

**目标**: 理解系统的业务逻辑和用户角色

**核心内容**:
- **角色分析** (`roles.md`): 用户角色、职责、权限矩阵
- **流程分析** (`workflows.md`): 业务流程、泳道图、步骤表
- **状态机分析** (`status-machines.md`): 状态定义、流转图、转换规则

**关键问题**:
- 系统有哪些用户角色？各自职责是什么？
- 核心业务流程如何流转？涉及哪些角色？
- 业务对象有哪些状态？如何转换？

---

### 技术维度考古 → `references/technical.md`

**目标**: 理解系统的技术实现

**核心内容**:
- **后端代码** (`backend.md`): 代码结构、分层、API端点
- **前端代码** (`frontend.md`): 页面组件、路由、状态管理
- **数据库** (`entities.md`, `schema.sql`): 表结构、实体关系、枚举

**关键问题**:
- 代码如何分层？有哪些业务模块？
- 前后端如何交互？使用什么技术栈？
- 数据如何存储？表之间如何关联？

---

### 接口维度考古 → `references/api.md`

**目标**: 理解系统的API接口

**核心内容**:
- **API端点** (`endpoints.md`): 接口清单、请求/响应格式
- **错误码** (`error-codes.md`): 错误码体系、命名规则
- **验证规则** (`validation-rules.md`): 参数验证、业务约束

**关键问题**:
- 有哪些API模块？每个模块提供什么功能？
- 错误码如何分类？如何处理？
- 接口参数有什么验证规则？

---

### UI维度考古 → `references/ui.md`

**目标**: 理解系统的用户界面

**核心内容**:
- **页面清单** (`pages.md`): 页面结构、路由映射、功能描述
- **UI规范** (`ui-spec.md`): 颜色、字体、间距、组件规范
- **截图收集** (`screenshots/`): 按角色/流程组织截图

**关键问题**:
- 系统有哪些页面？如何导航？
- 使用什么设计规范？
- 页面交互如何设计？

---

### 运维维度考古 → `references/ops.md`

**目标**: 理解系统的运维和部署

**核心内容**:
- **操作手册**: 用户操作流程、常见问题
- **测试报告**: 测试环境、测试账号、已知问题
- **环境配置**: 部署架构、配置文件、依赖服务

**关键问题**:
- 系统如何部署？
- 有哪些测试账号？
- 常见问题如何解决？

## 考古技巧

### 信息来源优先级
1. **代码**: 源代码中的注释、结构、命名
2. **配置**: 配置文件中的环境、依赖、参数
3. **文档**: 现有文档、Wiki、需求文档
4. **人员**: 业务人员、开发人员、运维人员
5. **运行中**: 日志、监控、数据库数据

### 分析方法
- **自顶向下**: 从业务流程 → 功能模块 → 代码实现
- **自底向上**: 从数据库表 → 代码逻辑 → 业务含义
- **由外向内**: 从UI界面 → API接口 → 后端逻辑

### 常见陷阱
- 只关注正常流程，忽略异常处理
- 只看代码不看配置和文档
- 忽略状态转换的约束条件
- 混淆不同环境的配置差异
- 截图不完整或命名不规范
