# 接口维度考古

## 概述

接口维度考古专注于梳理系统的API接口，包括端点定义、请求响应格式、错误码等。

## 1. API端点分析 (endpoints.md)

### 分析目标
- 梳理所有API端点
- 理解请求和响应格式
- 识别接口权限要求

### 输出模板

```markdown
# API 接口清单

## API 概览

- **Base URL**: `/api/v1`
- **认证方式**: JWT Bearer Token / OAuth2 / API Key
- **响应格式**: JSON
- **字符编码**: UTF-8

### 统一响应结构

```json
{
  "code": 200,
  "message": "success",
  "data": {},
  "timestamp": 1234567890
}
```

### HTTP状态码映射

| HTTP状态 | 说明 | 业务场景 |
|----------|------|----------|
| 200 | 成功 | 请求成功处理 |
| 400 | 请求错误 | 参数校验失败 |
| 401 | 未认证 | Token过期/无效 |
| 403 | 无权限 | 权限不足 |
| 404 | 不存在 | 资源不存在 |
| 500 | 服务器错误 | 系统内部错误 |

---

## 1. 模块接口 (`/module-name`)

### 1.1 接口名称

- **URL**: `POST /api/v1/module-name/action`
- **权限**: `ROLE_CODE`
- **描述**: 接口功能说明

**请求参数**:
```json
{
  "field1": "string",
  "field2": 123,
  "field3": {
    "nested": "value"
  }
}
```

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|:----:|------|
| field1 | String | ✅ | 参数说明 |
| field2 | Integer | ❌ | 参数说明 |

**响应示例**:
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "result": "value"
  }
}
```

**错误响应**:
```json
{
  "code": "ERROR_CODE",
  "message": "错误描述",
  "data": null
}
```
```

### 关键问题
- 有哪些API模块？每个模块提供什么功能？
- 接口如何组织？按资源？按功能？
- 认证如何实现？
- 分页如何处理？
- 排序和筛选如何处理？

---

## 2. 错误码分析 (error-codes.md)

### 分析目标
- 梳理所有业务错误码
- 理解错误码的分类和命名规则
- 建立错误码与场景的映射

### 输出模板

```markdown
# 业务错误码清单

## 错误码体系说明

### 命名规范
```
{模块}_{序号}
模块: AUTH/CUST/FIN/BID/LOAN/DOC/QRY/...
序号: 三位数字，如 001, 002
```

### 错误响应结构
```json
{
  "code": "ERROR_CODE",
  "message": "错误描述",
  "data": null
}
```

---

## 一、通用错误码

| 错误码 | 错误信息 | HTTP状态 | 说明 |
|--------|----------|----------|------|
| SUCCESS | 操作成功 | 200 | 成功响应 |
| BUSINESS_ERROR | 业务处理异常 | 400 | 通用业务错误 |
| RESOURCE_NOT_FOUND | 资源不存在 | 404 | 资源不存在 |
| SYSTEM_ERROR | 系统内部错误 | 500 | 系统异常 |
| UNAUTHORIZED | 未登录或登录已过期 | 401 | 认证失败 |
| FORBIDDEN | 无权限访问 | 403 | 权限不足 |
| PARAM_ERROR | 参数校验失败 | 400 | 参数验证失败 |

---

## 二、认证模块错误码 (AUTH)

| 错误码 | 错误信息 | HTTP状态 | 触发场景 |
|--------|----------|----------|----------|
| AUTH_001 | 登录账号或密码错误 | 400 | 登录验证失败 |
| AUTH_002 | 邀请码不存在 | 400 | 客户注册时邀请码验证 |
| AUTH_003 | 邀请码已使用或已过期 | 400 | 邀请码状态检查 |

### 异常抛出示例

```java
// Java
throw new BusinessException("AUTH_001", "登录账号或密码错误");

// Python
raise BusinessError("AUTH_001", "登录账号或密码错误")

// JavaScript
throw new BusinessError("AUTH_001", "登录账号或密码错误")
```

### 前端处理示例

```typescript
if (error.code === 'AUTH_001') {
  message.error('登录账号或密码错误');
} else if (error.code === 'CUST_002') {
  message.error('客户状态不正确，无法审批');
}
```
```

### 关键问题
- 错误码如何分类？按模块？按级别？
- 错误码命名有什么规则？
- 错误信息是否国际化？
- 是否有错误码的文档或注释？
- 前端如何处理错误码？

## 3. 验证规则分析 (validation-rules.md)

### 分析目标
- 梳理接口参数验证规则
- 理解数据格式要求
- 识别业务约束条件

### 输出模板

```markdown
# 接口验证规则清单

## 验证规则概览

| 规则类型 | 说明 | 示例 |
|----------|------|------|
| 必填 | 字段不能为空 | @NotNull |
| 格式 | 字段格式校验 | @Email, @Pattern |
| 长度 | 字符串长度限制 | @Length, @Size |
| 范围 | 数值范围限制 | @Min, @Max |
| 业务 | 业务规则校验 | 自定义验证器 |

---

## 1. 通用验证规则

| 字段类型 | 验证规则 | 错误提示 |
|----------|----------|----------|
| 手机号 | 11位数字，1开头 | "手机号格式不正确" |
| 邮箱 | 标准邮箱格式 | "邮箱格式不正确" |
| 身份证 | 18位身份证号 | "身份证号格式不正确" |
| 金额 | 正数，最多2位小数 | "金额格式不正确" |
| 日期 | yyyy-MM-dd | "日期格式不正确" |

---

## 2. 业务字段验证

### 客户相关

| 字段 | 验证规则 | 说明 |
|------|----------|------|
| enterpriseName | 必填，1-200字符 | 企业名称 |
| registeredCapital | 必填，非负数 | 注册资本 |
| legalPerson | 必填，1-50字符 | 法人代表 |

### 融资相关

| 字段 | 验证规则 | 说明 |
|------|----------|------|
| financingAmount | 必填，>0，<=资产总额80% | 融资金额 |
| financingTerm | 必填，1-36个月 | 融资期限 |
| interestRateMin | 必填，0-100% | 利率下限 |
| interestRateMax | 必填，>=下限，<=100% | 利率上限 |

---

## 3. 状态转换验证

| 当前状态 | 允许转换到 | 验证条件 |
|----------|------------|----------|
| STATE_A | STATE_B | 条件1 && 条件2 |
| STATE_B | STATE_C | 用户有权限 |
```

### 关键问题
- 有哪些通用验证规则？
- 不同业务对象的验证规则有什么差异？
- 状态转换有什么约束条件？
- 验证失败如何返回错误信息？

## 考古方法

### 信息来源
- Swagger/OpenAPI 文档
- 代码中的注解（@NotNull, @Valid等）
- 前端API调用代码
- Postman collection
- 单元测试中的测试用例
- 错误日志

### 分析技巧
1. **从代码注解推断规则**: @NotNull, @Size, @Pattern等
2. **从测试用例推断规则**: 边界值测试、异常测试
3. **从前端验证推断规则**: 表单验证、输入框限制
4. **从错误日志推断规则**: 参数校验错误信息

### 工具辅助
- **Swagger UI**: 查看所有API文档
- **Postman**: 导入API collection进行测试
- **代码搜索**: 搜索@Valid, @NotNull等注解
- **抓包工具**: Fiddler/Charles抓取实际请求

### 常见陷阱
- 只看正常请求，忽略错误场景
- 忽略隐式验证（如数据库约束）
- 混淆前端验证和后端验证
- 忽略验证规则的优先级和顺序
