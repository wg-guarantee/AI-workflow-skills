# 业务维度考古

## 概述

业务维度考古专注于从业务角度理解遗留系统，包括用户角色、业务流程、状态流转等核心业务逻辑。

## 1. 角色分析 (roles.md)

### 分析目标
- 识别系统中所有用户角色
- 理解每个角色的职责和权限
- 梳理角色之间的协作关系

### 输出模板

```markdown
# 角色权限清单

## 角色概览

系统共定义N种用户角色：

| 角色 | 代码 | 职责 |
|------|------|------|
| 角色名 | ROLE_CODE | 简要职责描述 |

## 角色详细定义

### 1. 角色名 (ROLE_CODE)

**角色代码**: `ROLE_CODE`

**角色描述**: 角色的详细说明

**主要功能**:
| 功能 | 描述 |
|------|------|
| 功能名 | 功能描述 |

**菜单权限**:
- 菜单项1
- 菜单项2

**数据权限**:
- 数据访问范围说明

## 权限矩阵

### 操作权限

| 操作 | 角色1 | 角色2 | 角色3 |
|------|:------:|:------:|:------:|
| 操作名 | ✅ | ❌ | ✅ |

### 数据权限隔离

| 角色 | 隔离规则 |
|------|----------|
| 角色名 | 隔离规则说明 |
```

### 关键问题
- 系统有哪些用户角色？
- 每个角色可以执行哪些操作？
- 角色之间如何协作完成业务流程？
- 哪些数据对哪些角色可见/不可见？

## 2. 流程分析 (business-flow-analysis.md, workflows.md)

### 分析目标
- 梳理核心业务流程
- 识别流程中的关键节点和决策点
- 理解流程中的异常处理

### 输出模板

```markdown
# 业务流程清单

## 流程总览

| 流程编号 | 流程名称 | 参与角色 | 复杂度 |
|----------|----------|----------|--------|
| WF-001 | 流程名称 | 角色1, 角色2 | 高/中/低 |

## WF-XXX: 流程名称

### 流程图（ASCII艺术）

```
┌─────────────────────────────────┐
│              流程名称              │
├─────────────────────────────────┤
│                                 │
│  [开始状态]                      │
│       │                         │
│       ▼                         │
│  [步骤1]                        │
│       │                         │
│   ┌───┴───┐                     │
│   │       │                     │
│ 条件A   条件B                    │
│   │       │                     │
│   ▼       ▼                     │
│[状态A]  [状态B]                  │
│                                 │
└─────────────────────────────────┘
```

### 详细步骤

| 步骤 | 操作人 | 操作 | 系统响应 | 异常处理 |
|------|--------|------|----------|----------|
| 1 | 角色 | 操作描述 | 系统行为 | 异常处理 |

### 业务规则

- 规则1: 说明
- 规则2: 说明
```

### 关键问题
- 核心业务流程有哪些？
- 每个流程涉及哪些角色？
- 流程中哪些地方可能出错？如何处理？
- 是否有循环或回退逻辑？

## 3. 状态机分析 (status-machines.md)

### 分析目标
- 识别业务对象的状态定义
- 梳理状态之间的转换关系
- 理解状态转换的触发条件和约束

### 输出模板

```markdown
# 状态机清单

## 状态机概览

系统包含N个核心业务状态机：
1. 对象A状态机
2. 对象B状态机
3. 对象C状态机

## 1. 对象A状态机

### 状态定义

| 状态 | 代码 | 说明 |
|------|------|------|
| 状态名 | STATUS_CODE | 状态说明 |

### 状态流转图

```
[STATE_A] ──触发──> [STATE_B]
     │
     │ 触发
     ▼
[STATE_C]
```

### 状态转换规则

| 当前状态 | 触发事件 | 目标状态 | 操作角色 |
|----------|----------|----------|----------|
| STATE_A | 事件描述 | STATE_B | 角色代码 |
```

### 关键问题
- 哪些业务对象有状态？
- 每个状态表示什么含义？
- 状态如何流转？什么触发流转？
- 是否有不可逆的状态转换？
- 是否有循环状态？

---

## 考古方法

### 信息来源
- 业务需求文档
- 用户手册
- 测试用例
- 代码中的业务逻辑
- 数据库状态字段
- 与业务人员访谈

### 分析技巧
1. **从代码推断业务**: 状态枚举、条件分支、权限检查
2. **从数据库推断业务**: 状态字段、关联表、触发器
3. **从UI推断业务**: 页面流程、按钮状态、表单验证
4. **从日志推断业务**: 操作日志、状态变更记录

### 常见陷阱
- 只关注正常流程，忽略异常处理
- 假设所有状态转换都是合法的
- 忽略角色权限的细粒度控制
- 混淆业务状态和技术状态
